## Scope and method

## 1. Project in one page

LedgerShield is a stateful benchmark for evaluating agents on enterprise accounts-payable and payment-integrity workflows. Instead of asking a model to answer a single prompt, it gives the agent a partially observable case, a budget, a limited number of steps, a toolbox, and intervention actions. The agent must inspect invoices, policies, ledger entries, vendor records, email threads, and callback evidence, then submit a final decision such as `PAY`, `HOLD`, `NEEDS_REVIEW`, or `ESCALATE_FRAUD`.

The benchmark is designed to test more than answer correctness. It also scores:

- whether the agent used the right tools
- whether it gathered grounded evidence
- whether it resisted urgency and spoofing pressure
- whether it took appropriate interventions
- whether its confidence was calibrated
- whether its final decision led to a safe downstream outcome
- whether it complied with AP/SOX-style controls
- whether it reasoned at the campaign level for linked-invoice attacks

The environment is implemented as a POMDP-style OpenEnv/FastAPI application. The repository contains:

- the benchmark environment and API server
- curated fixtures for vendors, policies, receipts, POs, ledger records, and cases
- procedural case-generation logic for challenge, holdout, and contrastive-twin suites
- grading and reward-shaping logic
- baseline and LLM-powered agent runners
- benchmarking, validation, and report-generation scripts
- CI, Docker, and OpenEnv metadata

## 2. Benchmark structure and task families

The curated benchmark contains 21 public cases, distributed across five task families:

| Task | Curated cases | Primary skill being tested | Typical output payload |
| --- | ---: | --- | --- |
| Task A | 4 | Invoice extraction under OCR/visual access | extracted fields, line items, evidence map |
| Task B | 5 | Three-way match and policy checking | decision, discrepancies, policy checks, evidence |
| Task C | 4 | Duplicate invoice and bank-account anomaly detection | decision, duplicate links, fraud flags, evidence |
| Task D | 6 | Social engineering and payment-change fraud | decision, reason codes, policy checks, evidence, counterfactual |
| Task E | 2 | Campaign-level / linked-invoice fraud detection | decision, cross-invoice links, campaign signals, evidence, counterfactual |

The case IDs in `live_model_comparison.json` confirm the curated distribution:

- `CASE-A-001` through `CASE-A-004`
- `CASE-B-001` through `CASE-B-005`
- `CASE-C-001` through `CASE-C-004`
- `CASE-D-001` through `CASE-D-006`
- `CASE-E-001` through `CASE-E-002`

The data loader can expand this benchmark beyond the 21 curated cases:

- challenge cases are generated by mutating hard cases (`task_c`, `task_d`, `task_e`)
- holdout cases are generated by a separate seeded holdout generator
- benign twins can be generated for risky `task_d` and `task_e` cases to test calibration

By default:

## 3. Core mental model: what LedgerShield is really simulating

LedgerShield is simulating a payment review workflow where the agent never starts with full truth. The benchmark separates what is public from what is hidden.

Public state includes:

- the case instruction
- visible documents
- already revealed artifacts
- budget and step limits
- messages
- risk and investigation summaries derived from what has been observed so far

Hidden state includes:

- latent risk signals
- which actions are actually required to investigate the case properly
- which artifacts should exist but are not yet revealed
- how long certain interventions take
- portfolio and campaign context
- the vendor callback simulator state
- the pressure event to inject mid-episode
- the latent evidence graph
- the downstream outcome that would occur for each possible final decision

This matters because LedgerShield is not only checking whether the final label is correct. It is checking whether the agent discovered enough of the hidden picture in a justifiable, policy-aware way.

## 5. Canonical data model and schema rules

`models.py` defines the core Python-side contract for environment state and agent interaction.

Important aliases and structures:

`LedgerShieldState` is especially important. It is the environment's internal working memory and stores:

- episode metadata (`episode_id`, `case_id`)
- budget and step counters

- tool trace

- observed and hidden risk signals
- interventions taken

### 5.2 `server/schema.py`

`server/schema.py` is the normalization and validation backbone for the rest of the system.

It defines:

- canonical field keys
- allowed investigation actions
- allowed intervention actions
- allowed decisions
- discrepancy, fraud, policy, and outcome categories
- reason-code alias normalization

Key utilities include:

- `normalize_text`
- `normalize_id`
- `safe_float`
- `numeric_match`
- `bbox_iou`
- `token_overlap`
- `canonical_reason_codes`
- `is_intervention_action`
- `is_investigation_action`

This module is what makes the benchmark robust to small formatting differences. Scores do not depend only on exact string equality; numeric normalization, text normalization, token overlap, and reason-code aliasing are all used to keep grading realistic.

This module is the entry point for assembling the working database.

- normalizes vendor keys using canonical, display, and vendor-key aliases
- applies `_case_defaults` to every case

The `_case_defaults` helper fills in missing operational fields like:

- `budget_total`
- `max_steps`
- `difficulty`
- `benchmark_split`
- `due_date_days`
- `documents`
- `gold`
- `task_label`
- `contrastive_pair_id`
- `contrastive_role`
- `initial_visible_doc_ids`

### 6.3 Case anatomy

A curated case generally contains:

- a `case_id`
- a `task_type`
- difficulty and operational metadata
- user-facing instructions
- documents with OCR and layout information
- a `gold` section containing the benchmark answer and hidden truth

Document objects commonly include:

- `doc_id`
- `doc_type`
- page or region metadata
- `ocr` or `accurate_ocr` token streams
- bounding boxes

The `gold` section can include, depending on task:

- extracted `fields`
- `line_items`
- `decision`
- `discrepancies`
- `policy_checks`
- `evidence_targets`
- `reason_codes`
- `fraud_flags`
- `duplicate_links`
- `cross_invoice_links`
- `campaign_signals`
- `unsafe_if_pay`

### 6.4 Generated cases: attacks, variants, holdouts, twins

Procedural generation is spread across three modules:

- `server/attack_library.py`
- `server/case_factory.py`
- `server/evidence_graph.py`

`server/attack_library.py` defines 16 adversarial attack templates across categories like identity attacks, document attacks, process attacks, and more persistent compromise patterns. Each attack can inject:

- reason codes
- fraud flags
- discrepancies
- unsafe-if-pay semantics
- instruction suffixes
- severity changes

`server/case_factory.py` uses those attacks to:

- generate challenge variants
- generate holdout cases
- randomize surface details such as invoice numbers, dates, or bank references
- derive variant IDs
- generate benign twins for risky task D/E scenarios

The benign-twin logic is especially important for calibration testing. Instead of merely asking whether a model catches fraud, the benchmark can ask whether it also avoids false escalation when the surface pattern looks similar but the underlying case is clean.

### 6.5 Evidence graphs

`server/evidence_graph.py` introduces a latent evidence graph abstraction with:

- `GraphNode`
- `GraphEdge`
- `UnlockRule`
- `EvidenceGraph`

The graph is used to represent evidence dependencies and scenario structure more formally than a simple flat list of risk signals. It supports scenario generation for:

- safe cases
- bank-change fraud
- duplicate-invoice fraud

This graph is also used later in grading for graph-aware evidence grounding and, in generated cases, for dynamic required-action derivation.

## 7. Hidden world and episode lifecycle

### 7.1 Hidden-world assembly in `server/world_state.py`

`build_hidden_world(case)` constructs the environment's latent internals from the public case.

It produces:

- `case_snapshot`
- `case_seed`
- `hidden_risk_signals`
- `revealed_artifacts`
- `artifact_unlock_order`
- `pending_events`
- `intervention_status`
- `dynamic_documents`
- `pressure_event`
- `vendor_simulator_state`
- `required_actions`
- `required_artifacts`
- `portfolio_memory`
- `campaign_context`
- `intervention_latencies`
- `latent_outcomes`
- `artifact_templates`
- `latent_evidence_graph`

### 7.2 Required actions and artifacts

The environment does not treat every step equally. It calculates what a competent investigation should have done.

Examples:

- Task A expects `ocr` and `zoom`
- Task B expects `lookup_policy`, `lookup_po`, and `lookup_receipt`
- Task C expects `search_ledger` and `compare_bank_account`
- Task D expects email-thread, vendor-history, policy, bank-compare, and ledger work
- Task E expects the Task D-style investigation plus callback verification, duplicate review, security routing, and vendor freeze actions

Additional required actions can be injected by:

- evidence-graph unlock rules
- hidden signals like `bank_override_attempt`
- duplicate/campaign indicators
- spoofing and policy-bypass signals

Required artifacts are similarly derived from hidden signals. For example:

- callback or bank-approval artifacts become required when bank-takeover signals are present
- duplicate cluster reports become required for duplicate or threshold-evasion signals
- receipt and PO reconciliation artifacts become required for fulfillment or purchasing mismatches

### 7.3 Campaign context

`_campaign_context()` derives portfolio-level pressure and attack context, including:

- linked invoice count
- linked case count
- amount at risk
- manual review capacity
- business criticality
- queue pressure

If the case itself does not already specify the at-risk amount, the code derives it from visible invoice totals.

### 7.4 Pressure events

`server/pressure_events.py` provides dynamic adversarial pressure such as:

- urgent CFO messages
- second spoofed emails
- IT system alerts

Pressure events are scheduled primarily for harder risky cases and are designed to arrive around the middle of an episode. The benchmark then scores whether the agent remained disciplined or made an unsafe payment under pressure.

### 7.5 Callback simulation

`server/vendor_simulator.py` simulates callback verification outcomes. It models whether:

- the vendor confirms the request legitimately
- the vendor disputes a supposed bank change
- an attacker confirms the change
- nobody answers

The hidden vendor-simulator state is built from latent compromise cues in the case. The resulting callback artifact later matters for both outcome safety and grading.

## 8. Environment loop in `server/environment.py`

### 8.1 Reset

`reset()` does the heavy lifting of starting an episode:

1. Selects a case, either explicitly or through curriculum logic.
2. Builds the hidden world.
3. Initializes visible docs and internal state.
4. Resets rewards, done flags, milestones, and watchdog state.
5. Returns the public observation.

### 8.2 Action space

The formal action space is a dictionary with:

- `action_type`
- `payload`

The `payload` varies by tool. Example action families include:

- `lookup_po`
- `lookup_receipt`
- `search_ledger`
- `inspect_email_thread`
- `compare_bank_account`

### 8.5 Dense reward shaping

The environment uses dense shaping in addition to terminal grading.

Key constants:

- `SHAPING_GAMMA = 0.98`
- `SHAPING_SCALE = 0.35`
- `INFO_GAIN_BONUS = 0.08`

Milestone rewards:

- `first_risk_signal = 0.05`
- `callback_requested = 0.04`
- `all_required_actions = 0.06`
- `artifact_revealed = 0.03`

The environment also tracks:

- potential-based reward shaping
- novel signal discovery
- milestone completion
- cost/budget penalties
- terminal score from `score_submission()`

### 8.6 Step processing

Every `step(action)` roughly follows this flow:

1. Increment counters and compute the pre-step potential.
2. Validate the action and normalize its payload.
3. Dispatch the action as:
   - a tool call
   - an intervention
   - a final submission
4. Convert the raw result into a normalized step result.
5. Add signal-discovery bonuses.
6. Deduct the action cost from budget.
7. Advance pending events and reveal artifacts whose latency has expired.
8. Inject any scheduled pressure event.
9. Append the step to trajectory and tool trace.
10. Update watchdog state.
11. If a terminal decision was submitted:
   - simulate the downstream outcome
   - compute compliance and currency snapshots
   - run the full grading rubric
   - compute heuristic risk summaries
   - finalize reward and info payload
12. If max steps or budget are exhausted, truncate the episode.

The result payload returned to clients contains the public observation, reward information, and metadata useful for evaluation.

## 9. Investigation tools and intervention logic

### 9.1 Tool implementation details in `server/tools.py`

This file is one of the most implementation-heavy parts of the project. It contains both the public tool functions and the subtle heuristics that make the benchmark feel realistic.

Notable details:

- vendor matching uses alias and token-overlap logic rather than naive exact matching
- sender-domain checks infer alignment heuristically when an exact expected domain is not available
- email-thread payloads include sender profile, request signals, and quoted directives
- threads can be reconstructed from OCR if a fixture thread is missing
- fast OCR deliberately injects deterministic noise
- accurate OCR returns the clean token stream
- ledger search can inject challenge/holdout near-miss noise for unsafe generated suites

### 9.2 Investigation tools

| Tool | Purpose | Notable implementation detail |
| --- | --- | --- |
| `zoom_tool` | zoom into a document region | supports visual-style document inspection workflows |
| `get_doc_crop_tool` | crop a specific region from a document | pairs with OCR-heavy extraction flows |
| `ocr_tool` | run OCR on a doc or region | `fast` mode adds deterministic seeded noise; `accurate` mode is clean |
| `lookup_vendor_tool` | retrieve vendor master data | uses normalized alias-aware vendor indexing |
| `lookup_vendor_history_tool` | inspect prior vendor change/fraud history | can derive fraud/bank-change signals |
| `lookup_policy_tool` | fetch policy rules | used heavily in Tasks B, D, and E |
| `lookup_po_tool` | retrieve PO records | supports three-way matching and PO reconciliation |
| `lookup_receipt_tool` | retrieve receipt records | supports receiving/fulfillment verification |
| `search_ledger_tool` | inspect prior ledger activity | duplicate/campaign detection; alias-aware; near-miss capable in generated suites |
| `inspect_email_thread_tool` | read email thread context | reconstructs thread features from OCR when structured data is absent |
| `compare_bank_account_tool` | compare submitted bank account vs approved vendor account | may derive bank mismatch flags |

### 9.3 Signal derivation in `server/transition_engine.py`

Tool outputs feed back into the environment through `update_from_tool_result()`.

Mappings include:

- vendor history can produce `historical_bank_change_rejected` and `historical_fraud_event`
- ledger search can produce `duplicate_near_match`
- email-thread inspection can produce `sender_domain_spoof`, `bank_override_attempt`, `urgent_payment_pressure`, and `policy_bypass_attempt`
- bank-account comparison can produce `bank_account_mismatch`

This matters because the environment does not grant "progress" only for taking actions; it grants progress for surfacing new hidden signals.

### 9.4 Interventions

Interventions are handled through `handle_intervention()` in `server/transition_engine.py`.

| Intervention | Immediate effect | Artifact or side effect |
| --- | --- | --- |
| `request_callback_verification` | requests callback review | schedules `callback_verification_result` |
| `freeze_vendor_profile` | freezes vendor profile | direct operational action, no delayed artifact |
| `request_bank_change_approval_chain` | requests approval-chain review | schedules `bank_change_approval_chain` |
| `request_po_reconciliation` | requests reconciliation | schedules `po_reconciliation_report` |
| `request_additional_receipt_evidence` | requests more receipt evidence | schedules `receipt_reconciliation_report` |
| `route_to_procurement` | routes case operationally | no delayed artifact |
| `route_to_security` | routes case to security | no delayed artifact |
| `flag_duplicate_cluster_review` | requests duplicate cluster analysis | schedules `duplicate_cluster_report` |
| `create_human_handoff` | packages a handoff summary | stores a structured `handoff_packet` |

Intervention latency is modeled explicitly. For example, callback verification may take 2 case-clock steps in bank-takeover scenarios.

## 10. Outcome simulation, risk heuristics, compliance, and special subsystems

### 10.1 `server/outcome_simulator.py`

This module simulates what operationally happens after the final decision. Outcomes include:

- `unsafe_payment_released`
- `policy_breach`
- `safe_payment_cleared`
- `manual_review_created`
- `false_positive_operational_delay`
- `fraud_prevented`

It also computes portfolio-facing metrics like:

- loss prevented
- loss released
- delay hours
- manual-review minutes
- linked invoice count
- queue pressure

This is important because the benchmark is meant to score real-world consequence, not just label agreement.

### 10.2 `server/risk_rules.py`

This module turns case truth into interpretable risk summaries and heuristic penalties.

It defines:

- `HIGH_RISK_SIGNALS`
- `MEDIUM_RISK_SIGNALS`
- `derive_case_risk_signals()`
- `risk_bucket()`
- `compute_due_date_potential()`
- `assess_submission_risk()`

`assess_submission_risk()` is where many "did the agent behave dangerously?" penalties live. It punishes patterns like:

- unsafe `PAY` on risky cases
- overconfident false escalation
- missing obvious high-risk signals
- failing to request callback/bank checks when warranted
- ignoring callback artifacts
- unnecessary escalation actions on safe cases

### 10.3 `server/compliance_engine.py`

The compliance engine adds enterprise-control realism. It models 8 SOX/AP controls (`SOX-AP-001` through `SOX-AP-008`) and scores whether the trajectory and outcome respected them.

Important features:

- control applicability depends on task type, instructions, risk signals, and gold policy failures
- compliance is not binary; it is a weighted score
- `compliance_penalty()` can penalize severe failures, capped at `-0.30`
- grading then adds a smaller compliance adjustment term with `COMPLIANCE_ADJUSTMENT_WEIGHT = 0.05`

### 10.4 `server/currency_engine.py`

This subsystem adds multi-currency realism.

It implements:

- IBAN validation
- SWIFT validation
- currency conversion
- sanctioned-currency detection
- mismatch warnings
- aging-report generation

The engine includes:

- a static FX table
- sanctioned currency logic
- a spread of 50 basis points

Task A uses a specific currency-validation side channel in grading. If the extracted bank/currency fields are incorrect or malformed, the score is adjusted.

### 10.5 `server/vendor_simulator.py`

This module is the hidden callback engine. It creates the simulated response to callback verification requests and exposes a grading weight helper:

- `build_vendor_simulator_state()`
- `simulate_callback()`
- `get_callback_grading_weight()`

Callback interpretation is scored directly for fraud-focused tasks. The benchmark is not satisfied by an agent that merely asks for a callback; it also wants the agent to interpret the result correctly.

### 10.6 `server/pressure_events.py`

This module schedules and scores adversarial pressure. It includes:

- pressure-event templates
- `schedule_pressure_event()`
- `evaluate_pressure_resistance()`

Pressure-resistance scoring later feeds into both final grading and public state summaries.

### 10.7 `server/curriculum.py`

LedgerShield has a curriculum layer for repeated training/evaluation settings. It defines tiers:

- novice
- competent
- proficient
- expert

It tracks:

- exponential moving competence
- stagnation
- weakest task family

It can then:

- choose the next case strategically
- bias toward weak areas
- adjust budget and max-step allowances by tier

### 10.8 `server/dual_agent_mode.py`

This file adds a watchdog/auditor layer, effectively a dec-POMDP-style second perspective.

It defines:

- `WatchdogVerdict`
- `WatchdogObservation`
- `WatchdogState`

The watchdog tracks suspicion, can warn or veto, and produces a joint-scoring signal through `score_dual_agent_episode()`. This is one of the more novel research-oriented pieces in the repo because it models analyst/auditor interaction instead of a single monolithic agent.

## 11. Grading rubric in detail

### 11.1 Shared scoring constants

From `server/grading.py`:

- `TASK_SCORE_MIN = 0.01`
- `TASK_SCORE_MAX = 0.99`
- `DEGENERATE_EVIDENCE_CAP = 0.25`
- `TASK_E_DEGENERATE_EVIDENCE_CAP = 0.10`
- `COMPLIANCE_ADJUSTMENT_WEIGHT = 0.05`
- `CURRENCY_ADJUSTMENT_WEIGHT = 0.03`
- `TASK_E_LINK_GATE_THRESHOLD = 0.85`

Scores are always clamped into the open benchmark range via `strict_task_score()`.

### 11.2 Evidence scoring

Evidence is not scored as a pure string comparison. `_single_evidence_score()` combines:

- document ID match
- page match
- bounding-box IoU
- token overlap

Weights:

- 0.35 document match
- 0.15 page match
- 0.30 bbox IoU
- 0.20 token overlap

If the environment includes an evidence graph, evidence scoring can also receive a graph-aware grounding bonus by checking whether cited docs correspond to revealed critical nodes.

### 11.3 Counterfactual scoring

`counterfactual_score()` evaluates:

- structural conditional language
- decision/risk vocabulary
- evidence specificity
- depth
- optionally evidence-graph edge citations

Task E extends this further with document-reference and amount-reference checks across linked invoices.

### 11.4 Degenerate-submission checks

`_degenerate_submission_check()` penalizes low-effort submissions such as:

- empty evidence maps
- missing reason codes on fraud tasks
- missing counterfactuals on tasks D/E
- missing discrepancy payloads where they should have been present

### 11.5 Trajectory sub-scores from `server/trajectory_grading.py`

Trajectory grading provides:

- `calibration_score()`
- `investigation_score()`
- `intervention_score()`
- `efficiency_score()`
- `downstream_outcome_score()`
- `resolution_state_score()`

Important details:

- the intervention base score was intentionally tightened to `0.15`
- repeated identical actions are penalized
- long trajectories over 8 steps incur a length penalty
- risky cases reward intervention behavior such as callback verification, security routing, vendor freeze, and duplicate review
- safe cases penalize unnecessary interventions

## 13. Agent implementations and inference stack

### 13.1 `llm_utils.py`

This file provides LLM-utility helpers that make the rest of the agent code more robust:

- `parse_json_dict()` extracts JSON even from fenced code blocks or mixed text
- `create_json_chat_completion()` handles provider-specific quirks around token parameter names and JSON-object response formatting

### 13.2 `inference.py`: the main baseline/runtime agent

`inference.py` is the most important non-server file in the repo. It is the primary baseline agent and the canonical runtime harness used by report-generation scripts.

Important characteristics:

- it follows a strict stdout contract with `[START]`, `[STEP]`, and `[END]` markers
- it can run against a local in-process environment or a networked environment URL
- it contains a capability-profile system keyed by model name/score
- it uses both heuristics and LLM planning depending on the tier
- it includes submission repair and task-specific hardening

`ModelCapabilityProfile` controls:

- `tier`
- `plan_mode`
- `repair_level`
- investigation/intervention budget bonuses
- planning token budget
- decision token budget

The rough profile tiers visible in `live_model_comparison.json` are:

- `standard` for weaker models
- `strong` for mid-tier models
- `elite` for stronger models

Agent pipeline in `run_episode_with_env()`:

1. Reset the environment.
2. Bootstrap OCR over visible invoice docs.
3. For Task A, do the minimal extraction-focused flow and submit quickly.
4. For other tasks, build a collected-state summary from tool outputs and observations.
5. Generate investigation candidates.
6. Rank or LLM-select actions.
7. Run interventions when needed.
8. Wait out delayed artifacts if useful.
9. Build a task-specific final submission.
10. Repair/sanitize the submission if required.
11. Submit the decision and record the trace.

Task-specific submission builders in this file handle:

- field and line-item extraction for Task A
- three-way-match discrepancy handling for Task B
- fraud/duplicate submissions for Task C
- email-fraud reasoning for Task D
- campaign-level multi-invoice reasoning for Task E

### 13.3 Guardrails for Tasks C and D

`task_c_guardrails.py` and `task_d_guardrails.py` are critical hardening layers.

Task C guardrails:

- derive grounded duplicate and bank-risk findings from collected evidence
- add constructive "clean" evidence when the case is safe
- sanitize reason/fraud outputs to grounded facts
- optionally repair wrong-direction decisions

Task D guardrails:

- detect sender spoofing, bank override, urgency, policy bypass, threshold evasion, suspicious history, and duplicate patterns
- build grounded `ESCALATE_FRAUD` or safe `PAY` submissions
- add constructive safe evidence on benign cases
- sanitize reason codes and evidence while preserving benchmark-relevant misses
- repair wrong-direction decisions when necessary

These guardrails are a big part of why stronger agents in this repo can remain accurate without being allowed to output arbitrary hallucinated reason codes.

### 13.4 `inference_llm_powered.py`

This is a more explicitly LLM-first runner. It is intended to reveal stronger separation between weak and strong models and writes richer debug artifacts to `LEDGERSHIELD_DEBUG_ARTIFACT_DIR`.

Compared with `inference.py`, it is:

- less deterministic
- more directly LLM-driven in planning/decisioning
- designed for live model comparisons
- instrumented to report API usage totals

### 13.5 `inference_improved.py`

This file is an alternate / experimental inference path. It retains much of the same benchmark logic but represents a separate attempt at prompting, planning, and validation strategy. It is useful as a sandbox for agent changes without disturbing the main baseline runner.

### 13.6 `llm_judge_grader.py`

This file adds an optional LLM-as-judge layer. It can:

- score reasoning quality
- aggregate capabilities
- compare multiple agents
- validate grader signal

It is not the core benchmark grader, but it is useful for meta-evaluation and analysis.

## 16. Notable design choices and what they imply

### 16.1 The benchmark resists shortcutting

Many benchmarks can be gamed by output-format imitation. LedgerShield makes that harder by combining:

- hidden signals
- required-action coverage
- delayed artifacts
- evidence grounding
- policy checks
- downstream outcome simulation
- pressure resistance
- counterfactual scoring
- compliance adjustments

### 16.2 The benchmark rewards process, not just the answer

This is visible everywhere:

- trajectory grading
- intervention scoring
- resolution-state scoring
- watchdog scoring
- degenerate-submission penalties

### 16.3 Procedural generation is layered, not foundational

The benchmark starts from a hand-authored curated set and then expands it with attacks, holdouts, and twins. This gives the project both:

- a stable public benchmark
- a way to test generalization and calibration beyond memorized curated examples

### 16.4 Documentation is partially artifact-driven

The repo is intentionally set up so that generated benchmark artifacts can update:

- the README
- docs pages
- OpenEnv metadata

That is a strong sign that the benchmark is meant to be published and maintained as a living evaluation artifact, not just a local toy project.

## 19. Bottom line

LedgerShield is not just a collection of AP fraud cases. It is a carefully layered evaluation system with:

- curated and generated case suites
- hidden-state world modeling
- realistic tool and intervention workflows
- consequence-aware scoring
- policy and compliance constraints
- calibration and contrastive evaluation
- benchmark artifact publishing
- extensive regression testing

If you want to understand the repo operationally, the most important files to study after this document are:

1. `server/environment.py`
2. `server/world_state.py`
3. `server/grading.py`
4. `server/tools.py`
5. `inference.py`
6. `server/data_loader.py`
7. `server/case_factory.py`
8. `tests/test_ledgershield_env.py`

Those files collectively explain most of the benchmark's actual behavior.
